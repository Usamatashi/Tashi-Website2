# IFRS Integrated Accounting System – Architecture Improvements & Replit Prompt Pack

## Current Observation

Your uploaded project is currently mainly a marketing/website React project, not yet a full accounting ERP structure. However, your attached accounting design prompts clearly show you are planning a serious IFRS-based accounting system.

To make the accounting tabs fully integrated and IFRS-compliant, you should design the software around ONE CENTRAL ACCOUNTING ENGINE.

The biggest mistake many accounting systems make:

* modules work separately
* statements are calculated independently
* cash book does not fully integrate with GL
* inventory does not update COGS automatically
* receivables/payables do not affect statements instantly

Your system should instead work like:

# ONE SOURCE OF TRUTH

Every module:

* Sales
* Purchases
* Inventory
* Cash Book
* Banking
* Payroll
* Expenses
* Fixed Assets
* VAT/GST
* Loans
* Production

must ultimately generate:

1. Journal Entries
2. Ledger Movements
3. Trial Balance Updates
4. Financial Statement Updates

in real time.

---

# Recommended Core Architecture

## 1. Central Journal Engine (MOST IMPORTANT)

Every transaction in every module should create journal entries automatically.

Example:

### Sales Invoice

Dr Accounts Receivable
Cr Sales Revenue
Cr VAT Output

### Payment Received

Dr Cash/Bank
Cr Accounts Receivable

### Purchase Invoice

Dr Inventory/Expense
Dr VAT Input
Cr Accounts Payable

### Salary Posting

Dr Salary Expense
Cr Salary Payable

### Depreciation

Dr Depreciation Expense
Cr Accumulated Depreciation

DO NOT allow statements to calculate directly from module tables.

Everything must flow through:

Transaction → Journal → GL → Trial Balance → Statements

# IFRS-Compliant Integration Design

# 1. General Ledger (GL)

This should be the heart of the system.

## Must Include

* Double-entry validation
* Auto-balancing
* Immutable posting
* Draft vs posted vouchers
* Reversing journals
* Recurring journals
* Fiscal year close
* Opening balances
* Audit trail
* User timestamp tracking
* Source module references
* Transaction locking after close

## Important

Never allow direct editing after posting.
Use:

* reverse
* adjustment entry
* correction journal

This is IFRS/audit compliant.

# Recommended Replit Prompts For Improvements

# Prompt 1 – Central Accounting Engine

Build a centralized accounting engine where every module transaction automatically generates double-entry journal entries. All financial statements must derive from the general ledger and trial balance only. Prevent direct statement manipulation. Include posting validation, immutable journals, reversing entries, and audit trails.

---

# Prompt 2 – IFRS Financial Statements

Build IFRS-compliant financial statements including Statement of Financial Position, Statement of Profit or Loss, Statement of Changes in Equity, and Statement of Cash Flows using dynamic account mapping from the chart of accounts.

---

# Prompt 3 – Cash Flow Automation

Create an automatic Statement of Cash Flows generator. Every cash and bank transaction should classify into operating, investing, or financing activities. Generate both direct and indirect method cash flow statements automatically from journal entries.

---

# Prompt 4 – Integrated Inventory Accounting

Build perpetual inventory accounting using Weighted Average Cost. Every inventory purchase, sale, return, and adjustment must automatically update stock quantity, inventory valuation, cost of goods sold, and related journal entries.

---

# Prompt 5 – Accounts Receivable Integration

Create a fully integrated accounts receivable module with invoices, receipts, credit notes, customer aging, payment allocation, expected credit loss provisioning, and automatic general ledger postings.

---

# Prompt 6 – Accounts Payable Integration

Create an integrated accounts payable module supporting vendor bills, debit notes, supplier payments, withholding tax, aging reports, and automatic accounting entries.

---

# Prompt 7 – Fixed Assets IFRS Module

Build a fixed assets register supporting acquisition, depreciation, disposal, impairment, revaluation, and multiple depreciation methods. Automatically generate depreciation journals and fixed asset schedules.

---

# Prompt 8 – Multi-Currency Engine

Implement IFRS-compliant multi-currency accounting with transaction currency, base currency, exchange rate tables, realized and unrealized foreign exchange gain/loss calculations, and period-end revaluation.

---

# Prompt 9 – Dynamic Chart of Accounts

Create a configurable IFRS-based chart of accounts supporting hierarchical parent-child structure, account tags, financial statement mapping, cost centers, and department/project segmentation.

---

# Prompt 10 – Audit & Security

Implement immutable audit trails for all accounting transactions including user tracking, timestamps, IP logging, approval workflow, posting permissions, maker-checker approval, and accounting period locking.

---

# Prompt 11 – Reporting Engine

Build a financial reporting engine where all reports derive from ledger balances and support filters for date ranges, branches, projects, departments, currencies, and cost centers.

---

# Prompt 12 – Accounting Database Design

Design a PostgreSQL accounting database optimized for financial integrity with journal_batches, journal_lines, chart_of_accounts, ledger_balances, fiscal_periods, exchange_rates, inventory_movements, fixed_assets, tax_transactions, and audit_logs tables.

---

# Prompt 13 – Automatic Posting Rules

Create a posting rules engine where every business event automatically maps to predefined debit and credit entries. Allow configurable accounting rules for sales, purchases, inventory, taxes, payroll, depreciation, and banking transactions.

---

# Prompt 14 – Trial Balance Driven Statements

Refactor the system so all financial statements derive exclusively from trial balance balances generated from journal entries. Eliminate any direct calculations from sales, purchases, or inventory tables.

---

# Prompt 15 – Fiscal Year Closing

Build year-end closing functionality including retained earnings transfer, opening balances, period locking, adjustment journals, and comparative financial statements.

---

# Prompt 16 – Banking Reconciliation

Create bank reconciliation functionality with imported bank statements, unmatched transaction handling, reconciliation adjustments, and automatic bank ledger updates.

---

# Prompt 17 – Cost Center Accounting

Implement cost center and project accounting where every transaction can optionally allocate amounts to departments, branches, projects, or business units for profitability analysis.

---

# Prompt 18 – Tax Compliance

Build VAT/GST and withholding tax management with automatic tax postings, tax return summaries, invoice tax calculations, and tax ledger reconciliation.

---

# Prompt 19 – Performance Optimization

Optimize accounting queries for large transaction volumes using indexed ledger tables, materialized views, posting queues, transaction batching, and summary balance tables.

---

# Prompt 20 – ERP Integration Philosophy

Refactor the application into a true ERP architecture where all modules operate through a unified accounting and transaction engine. Ensure inventory, sales, purchases, banking, payroll, and financial statements remain synchronized in real time.

---

# Final Recommendation

The single most important thing:

DO NOT build accounting statements separately.

Build:

Unified Transaction Engine → Journal Engine → General Ledger → Trial Balance → Statements

If this foundation is correct:

* IFRS compliance becomes easier
* auditing becomes easier
* automation becomes easier
* reconciliation becomes easier
* scaling becomes easier
* multi-company becomes easier
* financial statements become reliable

This architecture is how professional ERP/accounting systems are designed.
